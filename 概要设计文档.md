## 1 引言

### 1.1 编写目的

本文档对 IronLog 健身记录应用进行概要设计，明确系统整体架构、模块划分、数据模型、接口约定及非功能性需求，为后续详细设计与开发提供依据。

### 1.2 项目背景

IronLog 以「训记」APP 的训练日志功能为基础，在其之上扩展以下能力：

1. **动作/器械 Wiki**：支持社区化的动作百科查询与编辑，附带评论、热度统计。
2. **个人健康档案**：导入 PDF 体检报告，构建用户健康画像。
3. **训练计划引擎**：支持按周、按日或机动模式编排练休分布。
4. **热量消耗估计**：基于运动类型、时长、体重等参数估算训练热量消耗。
5. **RM 与训练容量分析**：内置 RM 计算器，提供动作 RM 估算与训练容量统计曲线。

### 1.3 术语与缩写

| 术语 | 说明 |
|------|------|
| RM | Repetition Maximum，某动作在给定次数下能举起的最大重量 |
| 1RM | 单次最大重复重量 |
| 训练容量 (Volume) | 组数 × 次数 × 重量 |
| MET | 代谢当量，衡量运动强度的标准单位 |
| Wiki | 动作/器械的社区知识库条目 |
| 练休分布 | 训练日与休息日在时间轴上的排布方式 |

## 2 系统总体架构

### 2.1 架构风格

采用 **前后端分离的单体架构**。本应用以记录和查询为主，用户规模有限，无需高并发支撑，因此选择最精简的技术栈，优先保证易开发、易维护。

```
┌──────────────────────────────────────┐
│       前端（浏览器 / PWA）            │
│  React + TypeScript + Vite           │
│  TailwindCSS · Recharts · 日历组件   │
└──────────────────┬───────────────────┘
                   │ HTTPS (REST API)
┌──────────────────▼───────────────────┐
│         后端（单进程）                │
│  Python 3.11 + FastAPI + Uvicorn     │
│  SQLAlchemy 2.0 · PyMuPDF · 等       │
├──────────────────────────────────────┤
│  本地文件系统（uploads/）             │
└──────────────────┬───────────────────┘
                   │
            ┌──────▼──────┐
            │ PostgreSQL  │
            │   16.2      │
            └─────────────┘
```

**精简原则**：不引入 Redis、消息队列、对象存储服务、API 网关等中间件。所有功能在单进程内完成，文件存储使用本地磁盘，后台任务使用 FastAPI 内置的 BackgroundTasks。

### 2.2 技术选型

| 层次 | 技术方案 |
|------|----------|
| 前端 | React 18 + TypeScript + Vite（构建为 PWA，手机浏览器可安装到桌面） |
| UI 与图表 | TailwindCSS + shadcn/ui + Recharts |
| 后端 | Python 3.11 + FastAPI + Uvicorn |
| ORM 与迁移 | SQLAlchemy 2.0 + Alembic |
| 数据库 | PostgreSQL 16（本机已运行） |
| 认证 | PyJWT + bcrypt |
| 文件存储 | 本地文件系统（ `./uploads/` ） |
| PDF 解析 | PyMuPDF（文本型）+ PaddleOCR（扫描型） |
| 搜索 | PostgreSQL `to_tsvector` 全文检索 |

### 3.3 M2 训练日志模块

**职责**：记录每次训练的动作、组数、次数、重量等明细，提供补录、复制、分享功能。该模块继承训记的核心训练日志理念。

#### 3.3.1 数据结构

```
Workout（训练记录）
├── id, user_id, date, start_time, end_time
├── plan_template_id (nullable, 关联的计划模版)
├── note（训练备注）
├── mood（训练感受 1-5）
└── exercises: [WorkoutExercise]
       ├── exercise_id, sort_order, superset_group
       └── sets: [WorkoutSet]
              ├── set_number, weight, reps
              ├── unit (kg / lb)
              ├── duration_sec (用于计时类动作)
              ├── distance_m (用于有氧类动作)
              ├── rpe (自感用力度 6-10)
              └── is_warmup, is_dropset, is_failure
```

#### 3.3.2 功能列表

| 功能 | 说明 |
|------|------|
| 创建训练 | 空白创建或从模版创建 |
| 记录组数 | 逐组输入重量/次数/时长，支持 kg/lb 切换 |
| 超级组 | 多个动作绑定为一组轮替执行 |
| 训练计时 | 组间休息倒计时、训练总时长 |
| 补录与编辑 | 对历史任意日期进行补录或修改 |
| 复制训练 | 复制某次训练到指定日期 |
| 分享 | 生成训练报表图片 |

**关键接口**：

```
POST   /api/v1/workouts                创建训练
GET    /api/v1/workouts?month=2026-03  按月查询
GET    /api/v1/workouts/:id            训练详情
PUT    /api/v1/workouts/:id            编辑训练
DELETE /api/v1/workouts/:id            删除训练
POST   /api/v1/workouts/:id/copy       复制训练
POST   /api/v1/workouts/:id/share      生成分享图
```

### 3.4 M3 动作/器械 Wiki 模块

**职责**：提供动作百科与器械百科的查询、编辑、评论和热度排行。

#### 3.4.1 Wiki 条目结构

```
WikiArticle
├── id, slug, title, type (exercise / equipment)
├── category（胸、背、腿、肩、臂、核心、有氧、综合器械……）
├── target_muscles: [muscle_id]       主要目标肌群
├── secondary_muscles: [muscle_id]    次要参与肌群
├── content_md（Markdown 正文：动作描述、要点、常见错误、变体）
├── media: [图片 / GIF / 视频链接]
├── met_value（MET 值，用于热量计算）
├── view_count, like_count, comment_count
├── status (draft / published / archived)
├── revisions: [WikiRevision]         版本历史
│      ├── revision_id, author_id, diff, created_at
└── comments: [Comment]
       ├── id, user_id, content, parent_id (支持楼中楼)
       ├── like_count
       └── created_at
```

#### 3.4.2 功能列表

| 功能 | 说明 |
|------|------|
| 浏览与搜索 | 按分类筛选、关键词全文搜索、肌群筛选 |
| 条目详情 | 展示正文、图文/视频、目标肌群可视化人体图 |
| 编辑条目 | 具有编辑权限的用户可提交修改，保留版本历史 |
| 评论系统 | 楼中楼评论，点赞，举报 |
| 热度排行 | 按浏览量/点赞量/评论量排序，支持周榜/月榜 |
| 审核机制 | 管理员审核编辑内容后发布 |

**关键接口**：

```
GET    /api/v1/wiki/articles?type=exercise&q=卧推    搜索
GET    /api/v1/wiki/articles/:slug                   条目详情
POST   /api/v1/wiki/articles                         新建条目
PUT    /api/v1/wiki/articles/:slug                   编辑条目
GET    /api/v1/wiki/articles/:slug/revisions          版本历史
GET    /api/v1/wiki/articles/:slug/comments           评论列表
POST   /api/v1/wiki/articles/:slug/comments           发表评论
POST   /api/v1/wiki/articles/:slug/like               点赞
GET    /api/v1/wiki/ranking?period=week&sort=views    热度排行
```

#### 3.4.3 热度算法

综合热度分 = 0.4 × 归一化浏览量 + 0.35 × 归一化点赞量 + 0.25 × 归一化评论量，引入时间衰减因子 `decay = exp(-λ × Δdays)` 保证新鲜度。排行数据直接通过 SQL 查询计算，应用层使用内存缓存（TTL 1 小时）避免重复计算。

### 3.5 M4 训练计划模块

**职责**：创建训练计划模版，安排练休分布，支持按周/按日/机动三种编排模式。

#### 3.5.1 计划模式

| 模式 | 说明 |
|------|------|
| **按周循环** | 固定每周 N 天训练，指定周几练什么（如"周一胸、周三背、周五腿"），自动循环 |
| **按日循环** | 以 N 天为一个周期（如 Push-Pull-Legs-Rest 四日循环），不受星期约束 |
| **机动模式** | 不固定日期，用户按顺序依次执行模版，训练时手动选取下一个模版 |

#### 3.5.2 数据结构

```
TrainingPlan（训练计划）
├── id, user_id, name, description, color
├── mode: weekly | cyclic | flexible
├── cycle_length（cyclic 模式下周期天数）
├── is_active
└── templates: [PlanTemplate]
       ├── id, name, sort_order, color
       ├── schedule_rule
       │    ├── weekly: { day_of_week: [1,3,5] }
       │    ├── cyclic: { day_in_cycle: 1 }
       │    └── flexible: null（按顺序推进）
       └── exercises: [TemplateExercise]
              ├── exercise_id, sort_order
              └── note（自由文本备注，如"4组 8-12次，组间90秒"）
```

#### 3.5.3 功能列表

| 功能 | 说明 |
|------|------|
| 创建/编辑计划 | 选择模式，添加模版，为模版编排动作 |
| 计划本管理 | 多个计划本，归类模版 |
| 日历视图 | 按颜色展示计划安排，支持拖拽调整 |
| 安排到日期 | 将模版手动指派到特定日期 |
| 按模版训练 | 开始训练时可选择计划/模版；激活后动作选择列表**仅显示**模版内动作（过滤，非排序/置顶） |
| 首页快捷方式 | 今日计划区域展示当天应执行模版，一键跳转训练流程并自动激活模版过滤 |
| 计划进度追踪 | 显示当前计划完成率、连续打卡天数 |

**关键接口**：

```
POST   /api/v1/plans                            创建计划
GET    /api/v1/plans                            我的计划列表
PUT    /api/v1/plans/:id                        编辑计划
POST   /api/v1/plans/:id/templates              添加模版
PUT    /api/v1/plans/:id/templates/:tid         编辑模版
GET    /api/v1/plans/calendar?from=&to=         日历视图
POST   /api/v1/plans/templates/:tid/schedule    安排到日期
GET    /api/v1/plans/templates/:tid             获取模版详情（含动作列表）
```

### 3.6 M5 个人健康档案模块

**职责**：导入外部健康数据，建立并维护个人健康档案。

#### 3.6.1 数据来源

| 来源 | 导入方式 | 解析内容 |
|------|----------|----------|
| PDF 体检报告 | 用户上传 PDF 文件 | OCR + 规则引擎提取关键指标（BMI、血压、血糖、血脂、肝肾功能等） |
| 手动录入 | 表单输入 | 体重、体脂率、围度（胸围/臂围/腰围/腿围）、体检指标 |

#### 3.6.2 数据结构

```
HealthProfile（健康档案）
├── user_id
├── basic_info: { height, gender, birth_date }
├── measurements: [BodyMeasurement]       时序数据
│      ├── date, weight, body_fat_pct
│      ├── chest, waist, hip, arm_left, arm_right, thigh_left, thigh_right
│      └── source (manual / huawei / pdf)
├── medical_reports: [MedicalReport]
│      ├── id, date, file_url
│      └── indicators: { bmi, blood_pressure, blood_glucose, ... }
├── huawei_imports: [HuaweiDataImport]
│      ├── import_id, import_date, file_url
│      └── parsed: { steps[], heart_rate[], sleep[], sport_records[] }
└── goals: [HealthGoal]
       ├── type (weight / body_fat / measurement)
       ├── target_value, deadline
       └── status (active / achieved / abandoned)
```

#### 3.6.3 PDF 解析流程

```
用户上传 PDF
    │
    ▼
文件保存至本地 uploads/ 目录
    │
    ▼
PyMuPDF 提取文本
    │
    ├── 文本型 PDF → 直接提取
    └── 扫描型 PDF → PaddleOCR 识别
    │
    ▼
正则匹配提取指标（血压、血糖、血脂……）
    │
    ▼
结构化数据写入 medical_reports
    │
    ▼
用户确认/修正提取结果
```

解析过程通过 FastAPI BackgroundTasks 异步执行，用户上传后立即返回，解析完成后可查看结果。

**关键接口**：

```
POST   /api/v1/profile/measurements          录入体测数据
GET    /api/v1/profile/measurements?from=&to= 查询体测时序
POST   /api/v1/profile/medical-reports        上传体检 PDF
GET    /api/v1/profile/medical-reports/:id     查看解析结果
PUT    /api/v1/profile/medical-reports/:id     修正解析结果
GET    /api/v1/profile/overview                健康档案总览
POST   /api/v1/profile/goals                  设置目标
```

### 3.7 M6 数据分析模块

#### 3.7.1 RM 计算器

**算法**：提供多种经典公式供用户选择，默认 Epley 公式。

| 公式 | 表达式 | 特点 |
|------|--------|--------|
| Epley | 1RM = w × (1 + r / 30) | 较通用，适合一般力量训练 |
| Brzycki | 1RM = w × 36 / (37 − r) | 低次数时更准确，适合力量举/低次数测试 |
| Lombardi | 1RM = w × r^0.10 | 适合高重复次数的训练 |
| O'Conner | 1RM = w × (1 + 0.025 × r) | 最保守，适合适合康复训练或新手/老年人群 |

其中 w = 重量，r = 次数。

**功能**：

* 手动输入重量与次数，即时计算 1RM 及各次数对应重量表。
* 从训练日志中自动提取每个动作的历史最佳组，估算 e1RM 趋势。
* 动作 RM 进步曲线（折线图），支持多动作叠加对比。

#### 3.7.2 训练容量统计

| 统计维度 | 说明 |
|----------|------|
| 单次训练容量 | 某次训练的总容量 Σ(weight × reps) |
| 按动作统计 | 某动作在时间范围内的容量变化曲线 |
| 按肌群统计 | 汇总该肌群下所有动作的容量（需结合 Wiki 的肌群映射） |
| 按周/月汇总 | 每周/每月总容量、总组数、训练频次 |

图表类型：折线图（趋势）、柱状图（对比）、饼图（肌群占比）。

#### 3.7.3 热量消耗估计

**计算模型**：

```
热量 (kcal) = MET × 体重(kg) × 时长(h)
```

* 每个 Wiki 动作条目关联一个 MET 值（力量训练一般 3.0-6.0，有氧 6.0-12.0）。
* 根据训练日志中各动作的有效训练时长（组数 × 每组耗时 + 组间休息）计算。

**关键接口**：

```
POST   /api/v1/analysis/rm-calc              RM 计算器
GET    /api/v1/analysis/rm-history?exercise_id=&from=&to=
                                             动作 RM 趋势
GET    /api/v1/analysis/volume?group_by=exercise|muscle&from=&to=
                                             容量统计
GET    /api/v1/analysis/calories?from=&to=   热量消耗统计
GET    /api/v1/analysis/summary?period=week  综合训练摘要
```

```
     │      │view/like_cnt │      └──────────────┘
     │      │status        │
     │      └──────────────┘
     │
     │1   N ┌──────────────────┐
     │──────│BodyMeasurement   │
     │      │──────────────────│
     │      │date, weight      │
     │      │body_fat_pct      │
     │      │chest, waist, ... │
     │      │source            │
     │      └──────────────────┘
     │
     │1   N ┌──────────────────┐
     └──────│MedicalReport     │
            │──────────────────│
            │date, file_url    │
            │indicators (JSONB)│
            └──────────────────┘
```

**索引策略**：

* `workout`: (user_id, date) 复合索引，加速按月查询。
* `wiki_article`: GIN 索引支持全文搜索 (`to_tsvector`)。
* `body_measurement`: (user_id, date) 复合索引。
* `comment`: (article_id, created_at) 复合索引。

## 5 接口设计规范

### 5.1 通用约定

* 基础路径：`/api/v1/`
* 认证方式：Bearer Token (JWT)，置于 `Authorization` 头。
* 请求/响应格式：JSON，字符集 UTF-8。
* 分页：`?page=1&size=20`，响应包含 `total` 字段。
* 时间格式：ISO 8601（`2026-03-26T15:00:00+08:00`）。
* 错误响应：

```json
{
  "code": 40001,
  "message": "参数校验失败",
  "details": [
    { "field": "weight", "reason": "must be positive" }
  ]
}
```

## 6 非功能性需求

### 6.1 性能

本应用为个人/小团体使用的记录工具，无高并发需求。单进程 FastAPI + PostgreSQL 足以支撑。

| 指标 | 目标 |
|------|------|
| API 响应时间 | ≤ 500 ms |
| PDF 解析（后台） | ≤ 60 s |

### 6.2 安全性

* JWT 令牌认证。
* 密码 bcrypt 散列存储。
* Wiki 编辑需审核，评论支持举报。

### 6.3 兼容性

* 前端为响应式 Web 应用（PWA），支持主流桌面与移动浏览器。
* PDF 解析兼容文本型与扫描型体检报告。

## 9 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| PDF 体检报告格式多样，解析准确率不足 | 健康档案数据质量低 | 提供人工修正界面；积累样本后训练专用 OCR 模型 |
| Wiki 恶意编辑/垃圾评论 | 社区内容质量差 | 编辑审核 + 评论敏感词 + 用户举报 + 信誉分 |
| 离线/弱网环境数据丢失 | 用户训练记录丢失 | 本地 SQLite 缓存 + 冲突合并策略 |
| 热量消耗估算偏差大 | 用户信任度降低 | 明示"估算值"，持续校准 MET 参数，有心率数据时优先使用心率公式 |
